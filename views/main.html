Intro

Angular Validation provides an effective, performant, highly customizable yet with reliable defaults, validation tool for your Angular web-app.

Concepts:

This module approaches validation via three rule types:

  • Element:
    • Each element is passed to an element specific validator to ensure it will meet its criteria.
  • Group:
    • Elements can belong to groups. Though while in a group elements will stull be subject to their element validators, only once the group passes its validation criteria will it count towards the final form pass/fail.
  • Instance:
    • The instance rule is the term attached to the concept of "Can the user submit the form or not". The instance rule requires all validation criteria, both group and element, to be met.

Installation

Method One:

Clone this repo and copy/paste the files in /build to the directory of your choice in your app. Then reference them in your index.html

Method Two:

bower install ngValidation --save

Of course you still need to reference the files in your index.html

How to Use:

Default Implementation:
  1. Inject the validation module into your App's module declaration: 
    angular.module('MyApp', ['ng-validation'])
  2. Inject the validation module's service into a controller that will be using it: 
    angular.module('MyApp', []).controller('MainCtrl', function($scope, ngValidation) {})
  3. Inside the controller, include: 
    ngValidation.init();
  4. From here, if you wish a form to be validated, add the attribute 
    ng-validation
    to the root of the form/section containing the form fields.
Custom Processing

If you need more than the defaults, such as adding a scenario, follow these steps:

  1. To the same form/section you added 
    ng-validation
    to, add an attribute name: 
    ng-validation="myForm"
    . Please use only alpha-numeric characters.
  2. Once you've added the name, open up the controller where you called 
    ngValidation.init()
    and add your rules as follows: 
    ngValidation.add_rule('myForm', {options}).init();
    make sure the name of the rule is the same name you gave your form in the markup. And of course, dont forget to call 
    .init();
    When you're done adding all the rules you'll need.
  3. You can now give your form custom options:

Available options:

Methods
    • Name: rule
    • Type: function
    • Arguments (order matters):
      • Sel: DOM object of the root of the form
      • el: DOM object of the current event target
      • e: current event object
    • Scope (this): The root validation class
    • Description: If you want to completely ignore the default rule for determining whether or not a form is valid, add your own function here.
    • Example:
    • 
myForm: {
  rule: function(sel, el, e) {
    // custom processing here
  }
}
    • Name: validator
    • Type: function/mixed
    • Default: false
    • Description: Entry point for custom validation code that can effect validation success. This method *
      MUST
      * return a true/false value. This method *
      MUST
      * contain an object or function that correlates to the element's type. Please see the example below.
    • Scope (this): ngValidation Class
    • Available arguments (in order):
      • el: the DOM active validation element
      • e: the active validation event object
    • Example:
    • 
myForm: {
  validator: {
    radio: function(el, e) {
      if(el.checked) {
        return true;
      } else {
        return false;
      }
    }
  }
}
    • Name: on_load
    • Type: function
    • Default: noop
    • Description: Entry point for user-provided custom code execution that happens on page load
    • Scope (this): the root Validation class.
    • Example:
    • 
myForm: {
  on_load: function() {
    console.log(this);
  }
}
    • Name: ele_fail
    • Type: function
    • Default: noop
    • Description: Entry point for custom code on an ELEMENT validation failure
    • Scope (this): ngValidation Class
    • Available arguments (in order):
      • el: the DOM active validation element
      • e: the active validation event object
    • Example:
    • 
myForm: {
  ele_fail: function(el, e) {
    console.log(el, e);
  }
}
    • Name: ele_success
    • Type: function
    • Default: noop
    • Description: Entry point for custom code on an ELEMENT validation success
    • Scope (this): ngValidation Class
    • Available arguments (in order):
      • el: the DOM active validation element
      • e: the active validation event object
    • Example:
    • 
myForm: {
  ele_success: function(el, e) {
    console.log(el, e);
  }
}
    • Name: instance_fail
    • Type: function
    • Default: noop
    • Description: Entry point for custom code on INSTANCE validation failure
    • Scope (this): ngValidation Class
    • Available arguments (in order):
      • sel: the base DOM object for this form instance.
    • Example:
    • 
myForm: {
  instance_fail: function(sel) {
    console.log(el);
  }
}
    • Name: instance_success
    • Type: function
    • Default: noop
    • Description: Entry point for custom code on INSTANCE validation success
    • Scope (this): ngValidation Class
    • Available arguments (in order):
      • sel: the base DOM object for this form instance.
    • Example:
    • 
myForm: {
  instance_success: function(sel) {
    console.log(sel);
  }
}
    • Name: submit_fail
    • Type: function
    • Default: noop
    • Description: Entry point for custom code on submit validation failure
    • Scope (this): ngValidation Class
    • Available arguments (in order):
      • sel: the base DOM object for this form instance.
    • Example:
    • 
myForm: {
  submit_fail: function(sel) {
    console.log(el);
  }
}
    • Name: submit_success
    • Type: function
    • Default: noop
    • Description: Entry point for custom code on submit validation success
    • Scope (this): ngValidation Class
    • Available arguments (in order):
      • sel: the base DOM object for this form instance.
    • Example:
    • 
myForm: {
  submit_success: function(sel) {
    console.log(sel);
  }
}
    Properties:
    • Name: exempt
    • Type: array
    • Description: An array containing an indefinite number of indexes of strings that correlate to elements you wish to avoid validation processing. It uses a jQuery like selector syntax, just without the $().
    • Example:
    • 
myForm: {
  // This will ignore all elements with the class of ignore
  exempt: ['.ignore']
}
    • Name: scenarios
    • Type: array of objects
    • Description: An array containing the desired validation scenarios to use.
    • Example:
    • 
myForm: {
  // This will load the scenarios you wish to use (example is using the password scenario)
  scenarios: [{name: 'password', options: {ui: '.meter', acceptance: 'strong'}}]
}
    • Name: passed_toggle
    • Type: string
    • Default: 'passed-validation'
    • Description: CSS class name used to denote a passed element.
    • Example:
    • 
myForm: {
  passed_toggle: 'my-passed-class'
}
    • Name: failed_toggle
    • Type: string
    • Default: 'failed-validation'
    • Description: CSS class name used to denote a failed element.
    • Example:
    • 
myForm: {
  failed_toggle: 'my-failed-element'
}
    • Name: show_failed_elements
    • Type: boolean
    • Default: true
    • Description: Flag that determines whether or not to update the UI with the fail toggles.
    • Example:
    • 
myForm: {
  show_failed_elements: true
}
    • Name: show_passed_elements
    • Type: boolean
    • Default: true
    • Description: Flag that determines whether or not to update the UI with the pass toggles.
    • Example:
    • 
myForm: {
  show_passed_elements: true
}
    • Name: show_passed_on_load
    • Type: boolean
    • Default: false
    • Description: Flag that determines whether or not to update the UI with the pass toggles on page load.
    • Example:
    • 
myForm: {
  show_passed_on_load: true
}
    • Name: show_failed_on_load
    • Type: boolean
    • Default: false
    • Description: Flag that determines whether or not to update the UI with the fail toggles on page load.
    • Example:
    • 
myForm: {
  show_failed_on_load: true
}
    • Name: groups
    • Type: mixed
    • Default: false
    • Description: Object that specifies which elements to treat as part of a group.
    • Options:
      • name: name of the group (can be anything you want)
      • container: DOM element that holds the group elements as children. (optional)
      • required: Elements within the group that are required by the user
      • number_required: number of elements in the group that when considered passing will also pass the group validation.
    • Example:
    • 
myForm: {
  groups: [{name: 'checkboxes', container: '#/#checkboxes', required: 'input[type="checkbox"]', number_required: 1}]
}
    • Name: group_toggle
    • Type: string
    • Default: 'ng-validation-group
    • Description: CSS class name that gets added to the group container if scpecified
    • Example:
    • 
myForm: {
  show_passed_elements: true
}
    • Name: submit
    • Type: string/mixed
    • Default: false
    • Description: String used as query selector for DOM node to be used as submit button. If left as default the class will try to find a submit button of either input type="submit" or button type="submit". If none are found the validatio instance will fail early with an exception.
    • Example:
    • 
myForm: {
  submit: '#/#my-submit-button'
}
    • Name: events
    • Type: Object of arrays
    • Description: Object to serve as a mapper for event types => element types.
    • Default schema: 
      •                   
                    events: {
                      keydown: ['text', 'password', 'email', 'textarea', 'tel'],
                      keyup: ['text', 'password', 'email', 'textarea', 'tel'],
                      click: ['submit', 'checkbox', 'radio'],
                      change: ['checkbox', 'radio', 'select'],
                      paste: ['text', 'password', 'email', 'textarea', 'tel'],
                      blur: ['text', 'password', 'email', 'textarea', 'tel'],
                    }
    • Example:
    • 
myForm: {
  events: {
    // These are overridable just as any other option. *HOWEVER* they do carry one small caveat:
    // If you change an event mapping, to include or remove an element type, you must still include
    // the defaults as members of the array. For example, the defaults include textareas to validate
    // on a paste event. If I did not want this behavior I can totally remove it from the mapping, I just
    // have to include the others like so:
    paste: ['text', 'password', 'email', 'tel']
    // *REMEMBER* this applies to *ADDING* an element type as well.
  }
}

Form Example 5 - Scenarios:Password Strength

Outline

Password strength is deteremined by a scoring system. The score is calculated by the following point values:

  • Length: 1 point. Every character contributes to the total length of the password.
  • Uppercase: 3 points.
  • Number: 4 points.
  • Special: 6 points. Characters such as '@!+=&*^
The point values themselves are immutable, however the score considered to be passing can be set by custom options.

Must be 15 characters long and have an arbitrary passing score of 35:

Options:

            
// In the ngValidation injected controller:
ngValidation.add_rule('form_5', {
  scenarios: [
    {
      name: 'password', 
      options: {
        standards: {
          strong: {
            ui: 'The Force is strong with this one'
          },
          medium: {
            ui: 'Do or do not, there is no try'
          },
          weak: {
            ui: 'I find your lack of faith disturbing'
          }
        }, 
        ui: '#/#pw1', 
        acceptance: 'strong'
      }
    }
  ]
}).init();

Must be 8 characters long and have a passing score of 15

Options:

            
// In the ngValidation injected controller:
ngValidation.add_rule('form_5', {
  scenarios: [
    {
      name: 'password', 
      options: {
        standards: {
          strong: {
            ui: "Strong"
          },
          medium: {
            ui: 'Acceptable'
          },
          mostly_strong: {
            ui: 'Almost strong',
            score: 22
          },
          weak: {
            ui: 'Too weak'
          },
          mostly_weak: {
            ui: 'Almost there',
            score: 11
          }
        }, 
        ui: '#/#pw2', 
        acceptance: 'medium'
      }
    }
  ]
}).init();

Miscellaneous.

Performance:

Angular validate toutes being extremely DOM and user friendly in terms of implementation and experience.

You may notice on this page, however, that when deleting characters from a password field, the browser seems to hang or "lag" when deleting the last character in that field for close to a second.

While this performance hitch was difficult to track down, we can assure you that it is NOT angular-validate's fault. In fact, it is an old Chrome related defect that occurs when many elements are on a page; specifically elements that precede a password element:

Read more about this bug here.

Release History:
0.3.3

New Features:

  • Callback Entry Points have been added to the following events:
    • Element Fail
    • Element Success
    • Instance Fail
    • Instance Success
    • Submit Fail
    • Submit Success

Patches:

  • Password strength scenario uses a better, more accurate algorithm that also can accommodate an indefinite number of standards in any order.
  • Added version number to API layer in library.
The Future!

Version one hopes to 'angularize' more of the solutions to the problems that form validation presents:

  • A more declarative approach to instantiation and options.
  • Using angular directives and model binding to handle events.
  • Exposing values to $scope to offer more control and reveal more possibilities.
  • An XHR mode that doesn't rely on a submit button but rather a 'sync status' type of indicator.
  • Use of promises instead of callbacks, particularaly in XHR mode.

And many more!
